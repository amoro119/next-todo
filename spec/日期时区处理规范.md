# 日期时区处理规范

## 1. 背景与问题

本应用面向中国用户（Asia/Shanghai，UTC+8）。任务的截止日期在 Supabase 中以 UTC 存储，客户端（浏览器/Electron）运行在本地时区环境。

### 核心约定

> **UTC 午夜对齐**：用户选择「5月13日」为截止日，存储时转换为 UTC 前一天 16:00（即北京时间 00:00），即 `2026-05-12 16:00:00+00`。读取时必须将此 UTC 时间转换回 Asia/Shanghai 日期，才能正确还原为「5月13日」。

---

## 2. 数据库存储格式

| 格式 | 示例 | 出现场景 |
|------|------|----------|
| DB UTC（带时区偏移） | `2026-05-12 16:00:00+00` | Supabase 原始值，旧版 Dexie 存储 |
| ISO 8601（带时区） | `2026-05-12T16:00:00+00:00` | Supabase 同步后写入 Dexie 的新格式 |
| 纯日期 | `2026-05-13` | `normalizeTodo` 标准化后的展示层格式 |
| 旧版错误格式 ⚠️ | `2026-05-12 160000` | 历史 bug 遗留，冒号和时区被错误剥离 |

---

## 3. 历史 Bug 根因

### 3.1 写路径（`TodoModal.tsx` `cleanDate`）

旧代码在保存任务时，将正确的 DB UTC 格式通过 `.replace()` 转成了非标准格式：

```typescript
// ❌ 旧代码（已修复）
return localDateToDbUTC(date)?.replace(' 16:00:00+00', ' 160000') || null;
```

`"2026-05-12 160000"` 不含冒号和时区信息，`new Date("2026-05-12 160000")` 在所有标准解析器中均返回 `Invalid Date`。

### 3.2 读路径（`app/page.tsx` `formatDbDate`）

旧代码对 ISO 字符串做了简单的字符串截取：

```typescript
// ❌ 旧代码（已修复）
if (/^\d{4}-\d{2}-\d{2}T/.test(val)) return val.slice(0, 10);
```

`"2026-05-12T16:00:00+00:00".slice(0, 10)` 得到 `"2026-05-12"`，丢弃了时区信息，导致时区转换完全失效，任务显示在错误的日期（5月12日而非5月13日）。

---

## 4. 修复后的数据流

### 4.1 写路径（用户选择日期 → 存入 Dexie/Supabase）

```
用户选择 "2026-05-13"
    ↓ localDateToDbUTC()
"2026-05-12 16:00:00+00"   ← 正确的 DB UTC 格式
    ↓ 写入 Dexie / 同步到 Supabase
```

`localDateToDbUTC` 逻辑：将本地日期减一天，设定为 UTC 16:00，产生 `YYYY-MM-DD 16:00:00+00`。

### 4.2 读路径（从 Dexie 读取 → 展示层）

```
Dexie 存储值（任意格式）
    ↓ normalizeTodo() → formatDbDate()
"2026-05-13"   ← Asia/Shanghai 纯日期
    ↓ 传递给所有展示组件
```

`formatDbDate` 处理所有历史格式，统一输出纯日期字符串。

---

## 5. 关键函数规范

### 5.1 `formatDbDate`（`app/page.tsx`）

所有 `due_date`、`start_date`、`next_due_date` 必须经此函数标准化，在 `normalizeTodo` 中统一调用。

**处理顺序：**

1. 旧版错误格式 `YYYY-MM-DD 160000` → 修复为 `YYYY-MM-DD 16:00:00+00`，继续处理
2. 纯日期 `YYYY-MM-DD` → 直接返回（无时区信息，不做转换）
3. 其他所有格式（ISO、DB UTC、JS Date 字符串）→ `new Date()` 解析 → `Intl.DateTimeFormat('en-CA', { timeZone: 'Asia/Shanghai' })` 转换

```typescript
function formatDbDate(val: unknown): string | null {
  if (!val || typeof val !== 'string') return null;

  if (/^\d{4}-\d{2}-\d{2} 160000$/.test(val)) {
    val = val.replace(' 160000', ' 16:00:00+00');
  }
  if (/^\d{4}-\d{2}-\d{2}$/.test(val)) return val;

  const d = new Date(val);
  if (!isNaN(d.getTime())) {
    return new Intl.DateTimeFormat('en-CA', {
      timeZone: 'Asia/Shanghai',
      year: 'numeric', month: '2-digit', day: '2-digit',
    }).format(d);
  }
  return null;
}
```

### 5.2 `localDateToDbUTC`（`components/TodoModal.tsx`）

将用户选择的本地日期转换为 DB UTC 格式写入存储。

```typescript
function localDateToDbUTC(date: string | null | undefined): string | null {
  if (!date || !/^\d{4}-\d{2}-\d{2}$/.test(date)) return null;
  const [year, month, day] = date.split('-').map(Number);
  const d = new Date(Date.UTC(year, month - 1, day, 16, 0));
  d.setUTCDate(d.getUTCDate() - 1);
  const pad = (n: number) => n.toString().padStart(2, '0');
  return `${d.getUTCFullYear()}-${pad(d.getUTCMonth() + 1)}-${pad(d.getUTCDate())} 16:00:00+00`;
}
```

### 5.3 `cleanDate`（`TodoModal.tsx` `cleanTodoDates`）

保存任务时对所有日期字段做标准化，输出必须为 DB UTC 格式或 null。

**处理规则：**

| 输入格式 | 输出 |
|----------|------|
| ISO 8601 `YYYY-MM-DDTHH:mm:ss.sssZ` | 原样返回 |
| DB UTC `YYYY-MM-DD HH:mm:ss+00` | 原样返回 |
| 旧版错误格式 `YYYY-MM-DD 160000` | 修复为 `YYYY-MM-DD 16:00:00+00` |
| 纯日期 `YYYY-MM-DD` | `localDateToDbUTC()` 转换 |
| 其他可解析格式 | `new Date()` → UTC 分量 → `YYYY-MM-DD 16:00:00+00` |
| 无法解析 | `null` |

---

## 6. 禁止事项

- **禁止** 对含时区的日期字符串使用 `.slice(0, 10)` 截取日期部分
- **禁止** 使用 `date.getFullYear()` / `date.getMonth()` / `date.getDate()`（本地时区方法），应使用 `Intl.DateTimeFormat` 或 UTC 方法
- **禁止** 对 DB UTC 格式使用 `.replace()` 剥离冒号和时区后缀
- **禁止** 在展示组件内部直接做时区转换，必须依赖 `normalizeTodo` 输出的纯日期
